# 书签云同步方案评估

> 日期：2026-05-16
> 状态：方案选型 / 设计草案，未实现
> 范围：评估三种实现路线的工作量、复杂度、难点；推荐渐进式落地

---

## 一、当前数据现状（前置事实）

- **存储**：Zustand `persist` → `localStorage["bookmark-store"]`，`version: 3`
- **数据规模**：典型用户 100-500 条书签 + 元数据；500 条 ≈ 200KB JSON
- **数据结构**（v3）：
  ```ts
  Category { id, name, subCategories: SubCategory[] }
  SubCategory { id, name, parentId }
  Bookmark { id, title, url, description?, favicon?, coverImage?, tags?, isFavorite?, subCategoryId, createdAt }
  ```
- **现有 ID 模式**：`bm_<timestamp>_<random>` / `cat_<...>` / `sub_<...>` —— 已是字符串、含时间戳、可全局唯一，**对云同步友好**
- **去重规则**：同子分类内按 URL 去重；URL 在入库前已归一化（去 utm/gclid/末尾斜杠）
- **后端**：完全没有，所有 API 路由是无状态代理（fetch-meta / proxy-image / screenshot）

---

## 二、为什么云同步对这个项目"贵"

不是 CRUD 难，难在**多设备并发修改的合并语义**。每多一台设备，可能的冲突场景数翻倍。

具体来说有三件事：

1. **没有"服务端时间"概念** —— `createdAt` 是设备本地时间；用户两台设备时钟不同步会让"谁先谁后"判断错乱
2. **分类是层级结构** —— Bookmark 引用 `subCategoryId`；如果设备 A 删了某子分类、设备 B 在该子分类下加了新书签，合并时书签该挂哪里？
3. **删除是"消失"** —— 当前删除直接从数组移除；如果设备 A 已同步删除、设备 B 在删除前又编辑了同一条，简单的"按时间戳取最新"会让 A 的删除被 B 的编辑覆盖（删除应该胜过编辑还是反过来？产品决策）

这些是任何离线优先（offline-first）应用都要面对的问题，没有银弹。

## 三、三条实现路线对比

### 路线 A：**用户自带存储**（推荐起步）

用户在设置面板里粘贴一个**Gist token / WebDAV URL+密码** / **GitHub 个人 Repo token**，应用把整个 `bookmark-store` JSON 推上去 / 拉下来。

**架构**：
```
浏览器 ←→ 用户的 Gist / WebDAV / OneDrive ←→ 浏览器（另一台）
```

**关键点**：
- **不要做实时增量同步**，只做"显式触发的全量推/拉"+"自动定时备份"
- 冲突策略 = **基于版本号 + 用户决策**：本地有 v3、云端有 v5 → 弹对话框让用户选"用云端覆盖本地 / 本地覆盖云端 / 看差异手动合并"
- 不需要后端、不需要登录、不需要付费云服务

**工作量**：3-5 天
- Day 1：抽象 `SyncProvider` 接口（push / pull / getVersion）+ Gist 实现
- Day 2：版本号机制（local `localVersion`、push 后递增；pull 检查 `cloudVersion > localVersion`）
- Day 3：冲突对话框 + diff 视图（"云端新增 X，本地新增 Y，云端删除 Z..."）
- Day 4：WebDAV provider（备选给坚果云用户）+ 错误处理 + 加密选项（可选）
- Day 5：UX 打磨 + 自动备份调度 + 文档

**优点**：
- 零运维（不用买服务器）
- 隐私好（数据在用户自己的账户里）
- 增量价值最快（哪怕只支持 Gist 一种 provider，对 5% 重度用户已是质变）

**缺点**：
- 多设备**实时**同步体验差（需要用户每次手动点同步）
- 用户要先有 GitHub / 坚果云账号并配置 token，门槛略高
- 不支持冲突时的**自动合并**，只能"二选一覆盖"或手动 diff

---

### 路线 B：**自建后端 + Last-Write-Wins**（中等投入）

跑一个 Next.js API + 数据库（Supabase / Vercel KV / SQLite），用户用 magic link 登录，每次 mutation 后台同步。

**架构**：
```
浏览器 ←→ 我们的 API ←→ Postgres
       (登录 token)     (用户表 + 书签表)
```

**关键点**：
- **保持 localStorage 为唯一信源**，云端只是镜像；离线仍然能用
- 每条 mutation 带 `updatedAt` 字段；服务端只接受比当前 `updatedAt` 新的写入（LWW = Last Write Wins）
- 删除变成"软删除"（加 `deletedAt` 字段），避免"A 删了 → 拉到 B → B 没拉之前又编辑了 → A 同步上来覆盖 B 的编辑"

**工作量**：2-3 周
- Week 1：选数据库 + auth（Supabase 最快，自带 magic link）+ schema 设计 + 推/拉 API
- Week 2：客户端集成 —— 离线队列、网络重连后批量同步、错误重试、过期数据冲突处理
- Week 3：迁移工具（让现有用户把 localStorage 导入云端账户）+ 删除账户 + 限流 + 监控

**优点**：
- 接近"自动同步"的体验，用户感知不到同步动作
- 多端实时（5-10 秒延迟）
- 可以加增值功能（分享书签集、协作分类）

**缺点**：
- **要持续运维**（备份、监控、限流、滥用防护）
- **隐私边界变了**：用户得信任我们不会读他的书签
- 成本（即使免费起步，用户多了 Supabase 也要花钱）
- LWW 仍有问题：用户在 A 上改了 title，在 B 上同时改了 description，**两个都是 LWW，谁后到谁覆盖**，另一个修改丢失

---

### 路线 C：**CRDT（Yjs / Automerge）**（最重投入）

用 Yjs / Automerge 这种 CRDT 库重写 store，配 Yjs 自带的 y-webrtc / y-websocket / y-indexeddb provider。

**架构**：
```
浏览器 (Yjs Doc) ↔ Yjs Provider (WebSocket / WebRTC / OneDrive) ↔ 浏览器
                       ↕
                   IndexedDB 本地持久化
```

**关键点**：
- CRDT 保证**任意顺序的并发修改都能自动合并**，没有"冲突"概念
- 同一条书签的 title 字段在两个设备同时改、最后会按字符级合并（或保留两个版本）
- 真正的实时协作可能（多人同时编辑一个分类）

**工作量**：4-6 周
- Week 1：用 Yjs 重写数据层（`Y.Map<Category>` / `Y.Array<Bookmark>`），保留 Zustand 作为只读视图
- Week 2：迁移现有 store 数据到 Yjs Doc，处理 v3 → v4 schema 升级
- Week 3：Provider 接入（推荐 y-websocket + 我们自建的 relay server，或 y-webrtc 纯 P2P）
- Week 4：UI 适配（Yjs 的 observe 回调要把变化映射回 React 渲染）
- Week 5-6：边界情况（账号合并、文档迁移、性能调优）

**优点**：
- **真正的离线优先 + 多设备无冲突**
- 工程上是最"正确"的解
- 副产品：撤销/重做、版本历史几乎免费（Yjs 内置）

**缺点**：
- **学习曲线陡** —— Yjs 的心智模型和命令式 Zustand 完全不同
- 整个 store 要重写，所有依赖 `useBookmarkStore` 的组件都要审一遍
- bundle 增加 ~50KB（Yjs core + provider）
- 需要 relay server 或 STUN/TURN（webrtc）做发现，仍然有运维成本

## 四、对比矩阵

| 维度 | A: 用户自带存储 | B: 自建后端 LWW | C: CRDT |
|---|---|---|---|
| 工作量 | 3-5 天 | 2-3 周 | 4-6 周 |
| 实时性 | 手动触发 | 5-10s 延迟 | <1s |
| 离线可用 | ✅ | ✅ | ✅ |
| 冲突处理 | 用户二选一 | 字段级 LWW（可能丢字段） | 自动合并，无丢失 |
| 运维成本 | 零 | 中（DB + 监控） | 中-高（relay + DB） |
| 隐私 | 最好（用户自管） | 一般（信任我们） | 取决于 provider |
| 用户门槛 | 中（要配 token） | 低（magic link） | 低 |
| 我们的法律风险 | 几乎零 | 有（数据合规） | 有 |

## 五、推荐路线（个人观点）

**先 A 后 B/C**，理由：

1. **A 是 80/20 解** —— 重度用户（你这种）有 GitHub 账号，配 token 不是问题；普通用户用不上多设备同步，根本不会启用。三天工作量解决重度用户痛点。
2. **A 的代码是 B/C 的基础设施** —— 抽象 `SyncProvider` 接口、版本号、冲突对话框这些**B 和 C 也都需要**。先做 A，等用户量起来再决定要不要做 B
3. **B 的隐患** —— 你这个项目卖点是"零账号、本地优先、隐私友好"，加自建后端等于卖点之一变成卖点的反面，需要充分理由
4. **C 的成本不匹配** —— CRDT 适合 Notion / Figma 这种多人协作场景，单人多设备的场景上用 CRDT 是过度工程

## 六、路线 A 的详细设计（如果决定做）

### 6.1 抽象层

```ts
// lib/sync/types.ts
export interface SyncSnapshot {
  version: number          // 单调递增,每次 push 后 ++
  syncedAt: string         // ISO 时间戳
  data: {
    categories: Category[]
    bookmarks: Bookmark[]
    displaySettings?: DisplaySettings
  }
  schemaVersion: 3         // 对应 Zustand persist version
}

export interface SyncProvider {
  id: "gist" | "webdav" | "onedrive"
  name: string
  isConfigured(): boolean
  getRemoteSnapshot(): Promise<SyncSnapshot | null>  // 不存在时返 null
  putRemoteSnapshot(snapshot: SyncSnapshot): Promise<void>
  getRemoteMeta(): Promise<{ version: number; syncedAt: string } | null>  // 不下载全量,只取版本号
}
```

### 6.2 同步流程

```
Push(用户点"上传"):
  1. snapshot = { version: localVersion + 1, data: store.getState() }
  2. provider.putRemoteSnapshot(snapshot)
  3. localVersion = snapshot.version
  4. localStorage.setItem("sync-last-push", now)

Pull(用户点"下载" 或 应用启动):
  1. meta = provider.getRemoteMeta()
  2. if (meta.version <= localVersion) → "已是最新",不动
  3. else if (localVersion === lastPushedVersion && meta.version > localVersion):
       // 本地没有未提交修改,云端更新 → 直接覆盖
       snapshot = provider.getRemoteSnapshot()
       store.replaceState(snapshot.data)
  4. else:
       // 双方都有修改,弹冲突对话框
       showConflictDialog({ local, remote })
```

### 6.3 冲突对话框

3 个动作 + diff 摘要：
- **保留本地**（把 cloud 视为可丢弃）
- **使用云端**（本地修改丢弃，弹二次确认）
- **手动合并**：v1 只提供"导出本地为 JSON 文件 → 用编辑器合并 → 重新导入"的工作流（不在 UI 里做三方 merge）

### 6.4 Gist Provider 具体实现

- 用户在 GitHub Settings → Developer settings → Personal access tokens → 创建一个 `gist` scope 的 fine-grained token
- 应用里输入 token + 可选 Gist ID（不填则首次自动创建一个 private gist）
- API：`GET/PATCH https://api.github.com/gists/{id}`，文件名约定 `bookmark-store.json` + `display-settings.json`
- 加密选项（可选）：用户输入一个口令，本地 WebCrypto 加密后再上传，避免 GitHub 看到内容

### 6.5 触发时机

- **手动**：设置面板的"立即同步"按钮
- **自动**：
  - 应用启动后 idle 时 pull 一次
  - 每次写操作（addBookmark / importBookmarks 等）后 debounce 30 秒触发 push（节流避免 API 限流）
  - 用户离开页面时（`visibilitychange` → hidden）push 一次

### 6.6 边界场景

| 场景 | 处理 |
|---|---|
| Token 失效 | toast 提示用户去重配置；同步开关变红 |
| 限流（GitHub 5000/h） | 退避重试 + toast |
| schemaVersion 不匹配（云端是 v2，本地是 v3） | 跑本地的 migrate，pull 完再 push 一次升级版的 |
| 加密口令丢失 | 不可恢复，提示用户重新初始化 |
| 网络断开 | 排队，连上后继续 |

## 七、和"三级分类"的优先级

- **如果三级分类改造和云同步都想做**：先三级（动 schema 风险低）→ 再云同步（基于稳定 schema）
- **如果只做云同步**：当前 v3 schema 已经足够开始，不阻塞
- 路线 A 即使先做了，将来加三级分类只需要：bump `schemaVersion`、写一个 v3→v4 的 migrate 函数；不需要重做同步层

## 八、决策点（你需要回答）

1. **同步的目标用户**：是你一个人多设备用，还是要给所有用户提供？
   - 一个人多设备 → 路线 A 完全够，甚至可以做得更糙（写死你的 Gist 配置）
   - 公开服务 → 路线 A 是最低成本，B/C 才是正式做法
2. **是否允许引入"账号"概念**：A 不需要，B/C 通常需要
3. **是否能接受"手动点同步"**：能接受 → A；要求自动 → B/C
4. **预算**：A 零成本；B 每月 $0-25（Supabase 免费档可能够）；C 需要常驻 ws server

我的强烈推荐：**先路线 A**，3-5 天落地，看实际使用情况再决定要不要往 B/C 演进。
