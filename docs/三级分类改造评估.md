## 一、当前数据模型（两级）

```ts
Category { id, name, subCategories: SubCategory[] }
SubCategory { id, name, parentId }
Bookmark { id, title, url, ..., subCategoryId }
```

- 书签直接挂在 SubCategory 上
- 导入器对浏览器三级及以下文件夹做"扁平化 + `[文件夹名]` 前缀"处理
- URL 绑定：`?category=&sub=`
- Store version: 2

---

## 二、目标数据模型（三级）

```ts
Category { id, name, subCategories: SubCategory[] }
SubCategory { id, name, parentId, children?: SubSubCategory[] }
SubSubCategory { id, name, parentId }  // parentId 指向 SubCategory
Bookmark { id, title, url, ..., leafCategoryId }  // 可挂在 Sub 或 SubSub
```

**设计决策点**（需要你定）：
1. 书签只能挂在最深一级（SubSub），还是任意一级都能挂？
   - 推荐：任意一级都能挂（`leafCategoryId` 可以是 Sub 或 SubSub 的 id），否则用户必须为每个 Sub 创建至少一个 SubSub，体验差。
2. SubSubCategory 是嵌套在 SubCategory.children 里，还是平铺在 Category 顶层数组？
   - 推荐：嵌套（`SubCategory.children: SubSubCategory[]`），与现有 `Category.subCategories` 模式一致。
3. 字段名：`subCategoryId` 改名为 `leafCategoryId`？还是保持 `subCategoryId` 但语义扩展？
   - 推荐：改名为 `leafCategoryId`，避免歧义。

## 五、对"云同步"的影响分析

| 场景 | 影响 |
|---|---|
| **先三级 → 再云同步** | 云同步只需处理 v3 schema，一次性设计，无历史包袱。推荐。 |
| **先云同步 → 再三级** | 云端已有 v2 数据 + 多客户端。升级时需要：(1) 云端 schema 迁移脚本；(2) 客户端版本协商（新客户端读到旧数据要能升级，旧客户端读到新数据要能降级或拒绝）；(3) 冲突合并逻辑更复杂（三级路径 vs 两级路径的 merge）。工作量 ×2。 |
| **同时做** | 不推荐。两个大改动耦合在一起，debug 困难。 |

## 七、我的建议

**暂不做**，理由：
1. 当前"扁平化 + 前缀"已经能吸收浏览器的深层文件夹，用户不会丢数据
2. 三级分类的 UI 复杂度（尤其移动端 sidebar 抽屉里三层树）对小屏体验是负面的
3. 如果后续要做云同步，先做云同步（v2 schema 更简单、冲突合并更容易），等云同步稳定后再评估是否有真实用户需求驱动三级

**如果决定做**：先做三级，再做云同步。顺序反过来代价翻倍。
